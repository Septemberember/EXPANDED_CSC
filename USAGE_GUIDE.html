CSC Engine — Usage Guide

CSC Engine is a standalone Python library for Condition Sequence Coverage (CSC) test case generation. CSC is a coverage criterion that sits between branch coverage and full path coverage — it systematically explores condition branches while avoiding path explosion through ancestor-loop matching and infeasibility marking.

Version: 0.1.0  |  Python: 3.10+  |  Key Dependency: Z3 Solver 4.15.0+

1. Quick Start

1.1 CSC CLI (recommended)

Build the Java bridge JAR once, then run CSC generation for any Java file:

# One-time: build the Java instrumentation bridge
cd java_bridge && mvn clean package && cd ..

# Generate CSC test cases and keep trace/CCT artifacts
python3 csc_tool.py path/to/MyClass.java \
  --session my_session \
  --keep-artifacts

# With options:
python3 csc_tool.py MyClass.java --bootstrap "x=42" --max-iter 50 --output results.json
python3 csc_tool.py MyClass.java --mode original          # Original CSC (no expansion)
python3 csc_tool.py MyClass.java --no-auto-instrument     # Already instrumented

This command auto-instruments the Java source, compiles it, runs the CSC loop, writes trace JSONL files, and outputs generated test cases with CCT visualization.

Default result location: if --output is not provided, csc_tool.py writes generated test cases to csc_tmp/{session}/{ClassName}/testcases.json. This directory is the recommended CSC result directory for the Refined TBFV stage.

1.2 Refined TBFV After CSC

Refined TBFV is intentionally a second offline stage. It consumes the test cases, trace JSONL files, and CCT produced by csc_tool.py, then verifies each observed path against an FSF file. 

python3 refined_tbfv_tool.py \
  --csc-result-dir csc_tmp/my_session/MyClass \
  --fsf path/to/MyClass_fsf.txt

The report is written to csc_tmp/my_session/MyClass/refined_tbfv_report.json. If a path violates an FSF unit, the corresponding CCT leaf is marked in a separate fault view: MyClass_cct_tbfv_fault.dot. A read-only failure-localization report is also written to cct_failure_localization.json, with a matching risk view at cct_failure_localization_<strategy>.dot.

With --csc-result-dir, the TBFV tool infers testcases.json, run_log.jsonl, MyClass_cct.pkl, and the default report/fault-view paths from one directory.

1.3 Python API (library usage)

Use the Python API directly to integrate CSC into your own tools:

# Install
pip install -e .

# Python usage
from csc_engine import CCT, Condition, ConditionResult, parse_path_info

# Build a CCT from a program execution trace
cct = CCT(use_bounded_range=True, range_bound=200)
csc_path = [
    "n <= 1,False,!(n <= 1),1",
    "i > 1,True,i > 1,1",
]
cct.add_sequence(parse_path_info(csc_path), "tc_0")

# Find the next uncovered branch
var_types = {"n": "int", "return_value": "int"}
target = cct.check_for_csc(var_types)
if target:
    pc = cct.construct_path_constraint(target)
    print(f"Next branch path constraint: {pc}")

# Visualize the tree
cct.save_dot("cct.dot")  # renders cct.png if Graphviz is installed

3. Core Concepts

3.1 Condition Sequence Coverage (CSC)

CSC explores program paths by looking at the sequence of conditions evaluated along each path. Unlike full path coverage (which can explode exponentially), CSC uses two stopping rules to keep the problem tractable:

  1. Condition 1 — Ancestor-Loop Matching: Stop exploring a branch when an ancestor node in the CCT has the same condition on the opposite side already covered. This captures loop-like repetition without unbounded unrolling.
  2. Rule 2 — Infeasibility Marking: When Z3 proves a path constraint is UNSAT, mark the branch as infeasible (no input can reach it).

3.2 Condition Case Tree (CCT)

The CCT is a binary tree where:

  • Internal nodes represent a program condition paired with a loop count (condition_string, loop_count).
  • Left child = the False branch of the condition.
  • Right child = the True branch of the condition.
  • Leaf nodes store test case identifiers (e.g. "tc_0"), infeasibility markers ("X"), or range-excluded markers. Leaf nodes also carry a test_inputs dict mapping each test case ID to its actual input values (e.g. {"tc_0": {"x": 5}}), shown in tree printouts and DOT visualizations.
  • Truncated leaves appear when a condition's loop_count exceeds MAX_LOOP_BOUND (2). The remainder of the path is stored for potential promotion later.
Key detail: Condition.__eq__ and __hash__ are based only on (condition_string, loop_count). The line number and input constraint are excluded — this is what enables ancestor-loop matching to work correctly.

3.3 Original CSC vs. Expanded CSC

ModeConstructorBehavior at Ancestor Match
Original CSC CCT(use_bounded_range=False) Always skip — strict Condition 1
Expanded CSC CCT(use_bounded_range=True, range_bound=200) Check bounded-range Z3 satisfiability first; explore if SAT, skip if UNSAT

Expanded CSC additionally checks whether in-range Z3 solutions exist ([-range_bound, range_bound] for integer variables). This means loops can be explored more thoroughly when values within the bounded range can distinguish new paths.

Key Methods

CCT.add_sequence(sequence, test_case, is_expanded=False, skip_truncation=False, test_inputs=None) method

Append a condition sequence to the CCT (Algorithm 1 from the paper).

  • sequence: List[ConditionResult] — the condition sequence
  • test_case: str — identifier (e.g. "tc_0")
  • is_expanded: bool — mark nodes as created during expansion
  • skip_truncation: bool — bypass MAX_LOOP_BOUND truncation
  • test_inputs: dict — e.g. {"x": 5}, stored on the leaf node for display

CCT.check_for_csc(var_types=None) → Optional[List[ConditionResult]] method

DFS left-first search for the first uncovered branch. Returns the target sequence or None if CCT is full / not initialized.

CCT.mark_infeasible(sequence) method

Mark a proven-infeasible branch with "X" marker.

CCT.construct_path_constraint(sequence) → str method

Build the path constraint (logical && of all conditions) from a condition sequence.

CCT.discover_all_uncovered(var_types) → List[dict] method

Walk the entire CCT and find ALL uncovered branches at once. Each result dict has idx, inputs, and debug_path.

CCT.print_tree() method

Print the CCT structure to stdout (text representation).

CCT.save_to_file(filepath) / CCT.load_from_file(filepath) method

Persist/restore CCT via pickle. load_from_file handles backward compatibility with old pickle formats.

CCT.save_dot(filepath) / CCT.to_dot(name="CCT") method

Generate Graphviz DOT output. save_dot also attempts PNG rendering if dot is available.

5.4 Z3 Helpers

java_expr_to_z3(expr_str, var_types) function

Convert a Java-like boolean expression string to a Z3 expression. Handles:

  • Logical operators: && → And, || → Or, ! → Not
  • Type mapping: int → BitVec(32), float/double → Real, boolean → Bool
  • Char literals: 'c' → BitVecVal(ord(c), 32)
  • Type casts stripped: (int), (long), (short), (byte), (char)

solver_check_z3(z3_expr, vars_types=None) → str function

Solve a Z3 expression. Returns model string (e.g. "[n=10, i=5]") or "OK" if UNSAT.

parse_result(z3_result_str) → dict function

Parse Z3 model string into {"var_name": "value", ...} dict.

add_value_constraints(logic_expr, var_types) → str function

Add global domain bounds: int ∈ [-32768, 32767], char ∈ [32, 126].

add_bounded_range_constraints(logic_expr, var_types, bound=200) → str function

Add tight bounded-range constraints for int variables: [-bound, bound].

5.5 High-Level Orchestration

generate_tcs_by_csc(program, session_id="default", use_expanded=False) → Result function

Sequential CSC: Find one uncovered branch at a time, Z3-solve for inputs. Call repeatedly until status 5 (CCT_FULL) or 6 (CCT_NOT_INITIALIZED).

batch_discover(program, session_id="default", use_expanded=False) → dict function

Phase 1: Find ALL uncovered branches at once. Returns {"status": "ok", "branches": [...], "cct_full": bool}.

batch_verify_and_merge(program, T, D, branches, var_types, session_id="default", use_expanded=False, max_workers=4) → List[dict] function

Phase 3: Parallel compile+execute, sequential CCT merge + Z3 verify.

parse_class_name(java_code) → str function

6. Workflows

6.1 Sequential CSC (one branch at a time)

The original algorithm. Call generate_tcs_by_csc in a loop until the CCT is full:

from csc_engine import generate_tcs_by_csc, CCT_NOT_INITIALIZED

program = open("MyClass.java").read()
session = "my_session"

while True:
    result = generate_tcs_by_csc(program, session_id=session, use_expanded=True)

    if result.status == 6:  # CCT_NOT_INITIALIZED — need bootstrap
        # Run the program once with default inputs to populate the CCT
        # Then call generate_tcs_by_csc again
        break

    if result.status == 0:  # SAT — test case found
        print(f"New inputs: {result.counter_example}")
        # Execute the program with these inputs, get the execution path,
        # add it to the CCT, then loop again
        break

    if result.status == 1:  # INFEASIBLE
        print("Branch marked infeasible, continue...")
        continue

    if result.status == 5:  # CCT_FULL
        print("All branches covered!")
        break

6.2 Batch CSC (three-phase)

For faster throughput, use the batch pipeline:

  1. Phase 1 — batch_discover: Find ALL uncovered branches
  2. Phase 2 (manual) — Generate main methods for each branch
  3. Phase 3 — batch_verify_and_merge: Parallel execute + merge
from csc_engine import batch_discover, batch_verify_and_merge
from csc_engine.java_exec import parse_top_level_md_def

program = open("MyClass.java").read()
var_types = parse_top_level_md_def(program)

# Phase 1: discover
discovery = batch_discover(program, session_id="batch1", use_expanded=True)
print(f"Found {len(discovery['branches'])} uncovered branches")

# Build main methods for each branch (your code)
for b in discovery["branches"]:
    b["main_method"] = build_main_method(b["inputs"])  # your helper

# Phase 3: verify and merge
T = "true"           # precondition
D = "divisor != 0"   # danger condition (optional safety property)
results = batch_verify_and_merge(
    program, T, D, discovery["branches"], var_types,
    session_id="batch1", max_workers=4
)
for r in results:
    if r["status"] == 2:
        print(f"Branch {r['idx']}: COUNTEREXAMPLE = {r['counterexample']}")
    else:
        print(f"Branch {r['idx']}: SAFE")

6.3 Building from Execution Traces (Python-Only)

You don't need Java at all — build CCTs directly from CSC path strings:

from csc_engine import CCT, parse_path_info, java_expr_to_z3, solver_check_z3

cct = CCT(use_bounded_range=True, range_bound=200)
var_types = {"n": "int", "return_value": "int"}

# Add paths as they are explored
cct.add_sequence(parse_path_info([
    "n <= 1,False,!(n <= 1),1",
    "i > 1,True,i > 1,1",
]), "tc_0")

# Find uncovered branches and Z3-solve
target = cct.check_for_csc(var_types)
while target:
    pc = cct.construct_path_constraint(target)
    from csc_engine import add_value_constraints
    z3_expr = java_expr_to_z3(add_value_constraints(pc, var_types), var_types)
    result = solver_check_z3(z3_expr, var_types)
    if result == "OK":
        cct.mark_infeasible(target)
    else:
        print(f"Inputs for next path: {result}")
        # Execute program with those inputs, get new path, add to CCT...
        break
    target = cct.check_for_csc(var_types)

7. Java Instrumentation Bridge

The java_bridge/ directory contains a Maven project that instruments Java source code with transparent CSCTrace probes at condition, assignment, declaration, and return points. At runtime those probes append structured JSON Lines events to the trace file configured by -Dcsc.trace.file=.... The Python side reads this JSONL trace to reconstruct CSC paths and Refined TBFV formulas.

7.1 Execution Path Format

Instrumented Java programs write one JSON object per executed event:

{"type":"ASSIGN","line":5,"kind":"var_decl","target":"n","rhs":"tons","value":"10"}
{"type":"COND","line":7,"kind":"if","order":1,"expr":"n >= 500","value":false}
{"type":"COND","line":10,"kind":"while","order":1,"expr":"n > 0","value":true}
{"type":"ASSIGN","line":18,"kind":"assign","target":"n","rhs":"n - 1","value":"9"}
{"type":"RETURN","line":21,"target":"return_value","rhs":"fee","value":"41"}

Each COND is treated as a CSC condition. Compound expressions such as A && B or A || B are instrumented as separate condition probes where supported, so the CCT and Refined TBFV stages reason over condition sequences rather than only whole branch statements.

7.2 Writing Instrumented Java Programs

Usually you should not write probes manually. Let the Java bridge generate calls such as:

int n = csc.bridge.CSCTrace.assignObject(5, "var_decl", "n", "tons", tons);
if (csc.bridge.CSCTrace.cond(7, "if", 1, "n >= 500", n >= 500)) {
    fee = csc.bridge.CSCTrace.assignObject(8, "assign", "fee", "n * 5", n * 5);
}
return csc.bridge.CSCTrace.retObject(21, "return_value", "fee", fee);
Semantic transparency: CSCTrace.cond(..., exprValue) returns the boolean value it receives, and assignObject/retObject return the runtime value they receive. The probe records information while preserving the source program's execution result.

8. Z3 Constraint Solving

8.1 Type Mapping

Java TypeZ3 SortDomain Constraints
intBitVec(32)[-32768, 32767] (value) / [-bound, bound] (bounded-range)
float, doubleRealUnbounded
booleanBoolTrue / False
charBitVec(32)[32, 126] (printable ASCII)
String, List, Map, etc.—Skipped (not modeled)

8.2 Solver Configuration

  • Timeout: 5 seconds per query (solver.set("timeout", 5000))
  • Model completion enabled for Bool variables
  • Signed interpretation for int/char BitVec values

8.3 Expression Conversion Details

# Java → Python AST → Z3
"x > 5 && y < 10"
# 1. Replace: "x > 5 and y < 10"
# 2. AST parse → BoolOp(And, [Compare(x, Gt, 5), Compare(y, Lt, 10)])
# 3. Z3: And(x > 5, y < 10)

11. Complete Examples

11.1 Python-Only CCT Construction

#!/usr/bin/env python3
"""Build a CCT from CSC path strings and find uncovered branches."""
from csc_engine import (
    CCT, Condition, ConditionResult, parse_path_info,
    java_expr_to_z3, solver_check_z3, add_value_constraints,
)

cct = CCT(use_bounded_range=True, range_bound=200)

# Simulate paths through: if (n <= 1) return -1; while (i > 1) { ... }
path_tc0 = [
    "n <= 1,False,!(n <= 1),1",
    "i > 1,True,i > 1,1",
    "n % i == 0,True,n % i == 0,1",
]
cct.add_sequence(parse_path_info(path_tc0), "tc_n=6")

path_tc1 = [
    "n <= 1,False,!(n <= 1),1",
    "i > 1,True,i > 1,1",
    "n % i == 0,False,!(n % i == 0),1",
    "i > 1,False,!(i > 1),2",
]
cct.add_sequence(parse_path_info(path_tc1), "tc_n=5")

cct.print_tree()

# Find and solve
var_types = {"n": "int", "return_value": "int"}
target = cct.check_for_csc(var_types)
if target:
    pc = cct.construct_path_constraint(target)
    bounded = add_value_constraints(pc, var_types)
    z3_expr = java_expr_to_z3(bounded, var_types)
    result = solver_check_z3(z3_expr, var_types)
    print(f"Z3 model: {result}")

cct.save_dot("example_cct.dot")

11.2 Batch Discovery + Verification

#!/usr/bin/env python3
"""Full batch CSC pipeline."""
from csc_engine import batch_discover, batch_verify_and_merge
from csc_engine.java_exec import parse_top_level_md_def

program = open("examples/sample_programs/LargestProperDivisor.java").read()
var_types = parse_top_level_md_def(program)

# Phase 1: Find all uncovered branches
discovery = batch_discover(program, session_id="demo", use_expanded=True)
print(f"Found {len(discovery['branches'])} uncovered branches")

# Build main methods for each branch
for b in discovery["branches"]:
    inputs = b["inputs"]
    # Build a main method that calls the target with these args
    args = ", ".join(str(v) for v in inputs.values())
    b["main_method"] = f'public static void main(String[] args) {{\n'
    b["main_method"] += f'    int r = largestProperDivisor({args});\n'
    b["main_method"] += f'    System.out.println("RETURN_VALUE: largestProperDivisor() = " + r);\n'
    b["main_method"] += f'}}'

# Phase 3: Execute and merge
results = batch_verify_and_merge(
    program,
    T="true",
    D="true",
    branches=discovery["branches"],
    var_types=var_types,
    session_id="demo",
    max_workers=4,
)
for r in results:
    status = "SAFE" if r["status"] == 0 else f"CE: {r['counterexample']}"
    print(f"  Branch {r['idx']}: {status}")

12. CLI Tool: csc_tool.py

csc_tool.py automates the entire pipeline: instrument, compile, bootstrap, and CSC loop. It uses the Java bridge JAR for auto-instrumentation, then drives the CSC engine to discover all branches.

12.1 Prerequisites

# Creates: java_bridge/target/csc-bridge-0.1.0-jar-with-dependencies.jar

13. CLI Tool: refined_tbfv_tool.py

refined_tbfv_tool.py verifies the paths generated by CSC against a Functional Scenario Form (FSF). It does not execute Java again. Instead, it reads the generated test case JSON, each test case's trace JSONL file, and optionally the persisted CCT pickle.

13.1 FSF File Format

Each FSF unit has a scenario predicate T and expected postcondition D:

[fsf_positive]
T: n > 0
D: return_value > 0

[fsf_non_positive]
T: n <= 0
D: return_value <= 0

The canonical filename is ClassName_FSF.txt. The tool also accepts ClassName_fsf.txt, and --fsf can point to any explicit FSF file. If no FSF file is found, the default smoke-test FSF is T: true, D: true.

13.2 Usage

python3 refined_tbfv_tool.py [<csc_results.json>] [options]

Options:
  --class, --class-name NAME    Target Java class name
  --java-file FILE             Infer class name and var types from Java source
  --run-log FILE               Infer class name and var types from CSC run_log.jsonl
  --csc-result-dir DIR         CSC result artifact dir, e.g. csc_tmp/session/ClassName
  --class-dir DIR              Backward-compatible alias for --csc-result-dir
  --fsf FILE                   Explicit FSF file
  --fsf-dir DIR                Directory containing ClassName_FSF.txt or ClassName_fsf.txt
  --var-types SPEC             Extra types: "return_value:int,n:int" or JSON object
  --cct FILE                   CCT pickle to annotate
  --output FILE                Report JSON path
  --fault-output-dir DIR       Directory for TBFV fault DOT/PNG
  --no-annotate-cct            Do not mark CCT leaves or render fault view
  --fail-on-violation          Exit 1 if any path violates an FSF unit
  --fail-on-unsupported        Exit 1 if any verification result is unsupported

13.3 Recommended Invocation

The preferred CSC-to-TBFV handoff is directory based: 

python3 refined_tbfv_tool.py \
  --csc-result-dir csc_tmp/try3_refined_tbfv/Try3 \
  --fsf dataset/CSC_V2_dataset/fsf_dir/Try3_fsf.txt

When --csc-result-dir is provided, the tool uses these default paths:

ArtifactDefault path
Generated test cases<csc-result-dir>/testcases.json
Run metadata and variable types<csc-result-dir>/run_log.jsonl
CCT pickle<csc-result-dir>/<ClassName>_cct.pkl
TBFV report<csc-result-dir>/refined_tbfv_report.json
CSC-stage CCT stats<csc-result-dir>/cct_stats.json
TBFV-stage CCT stats<csc-result-dir>/cct_tbfv_stats.json
TBFV fault DOT<csc-result-dir>/<ClassName>_cct_tbfv_fault.dot
Failure localization report<csc-result-dir>/cct_failure_localization.json
Failure localization DOT<csc-result-dir>/cct_failure_localization_statement_presence.dot by default when traces are available

DOT files are not rendered by default. To inspect the CCT views visually, run the generated render_cct.sh script in the result directory; it renders the normal CCT, the TBFV fault view, and all cct_failure_localization_*.dot strategy views when they exist.

Passing <csc_results.json> directly is still supported, especially when results come from a non-standard generator. In that mode, use --run-log, --cct, --class, or --var-types as needed to provide missing metadata.

13.4 How Trace Files Are Found

Refined TBFV finds the trace for each generated test case through the trace_path field stored in testcases.json. This is true for both sequential and batch CSC runs. 

{
  "iteration": 2,
  "branch_idx": 5,
  "strategy": "batch",
  "trace_path": "csc_tmp/demo/Try3/traces/round_02/b5/trace.jsonl"
}

The verifier does not derive this path from the test case ID. It loads the record, reads record["trace_path"], parses that JSONL file, then reconstructs Ct_in and wp(path, D_i) from the actual executed events. If a record has no trace_path, that path/FSF pair is reported as skipped with reason missing_trace_path.

13.5 Verification Formula

For every observed path and every FSF unit, the tool first checks scenario matching:

SAT(T_i && Ct_in)

If the path intersects the scenario, it verifies that no counterexample exists:

UNSAT(T_i && Ct_in && !wp(path, D_i))
  • Ct_in is the input-state path condition reconstructed from condition events.
  • wp(path, D_i) is the expected postcondition rewritten backward through value events.
  • A SAT result for the counterexample formula means the path violates the FSF unit.

13.6 Report and Fault View

The report summary counts results across all generated test cases and all FSF units:

{
  "summary": {
    "testcases": 4,
    "fsf_units": 4,
    "passed": 4,
    "failed": 0,
    "skipped": 12,
    "unsupported": 0
  },
  "results": [...]
}

When --cct is provided, every failed result is added to the matching CCT leaf via TBFV-specific metadata. The normal CSC CCT view is left unchanged; the fault distribution is exported separately as *_cct_tbfv_fault.dot. The same annotated CCT is then analyzed to produce cct_failure_localization.json.

13.7 Stage-Boundary CCT Statistics

CCT statistics are collected with one full tree traversal only at stage boundaries: after CSC generation and after Refined TBFV fault annotation. They are not recomputed after every iteration or every branch merge.

14. CLI Tool: failure_localization_tool.py

failure_localization_tool.py is a read-only analysis tool for an already annotated CCT. It does not run Java, does not re-run TBFV, and does not mutate the CCT. Its input is a CCT where Refined TBFV has marked failing leaves; its output is a JSON ranking report plus a DOT risk view. 

python3 failure_localization_tool.py \
  --csc-result-dir csc_tmp/my_session/MyClass

# Or point directly to a CCT pickle:
python3 failure_localization_tool.py \
  --cct csc_tmp/my_session/MyClass/MyClass_cct.pkl \
  --output csc_tmp/my_session/MyClass/cct_failure_localization.json

Options:

--csc-result-dir DIR   CSC result artifact dir, e.g. csc_tmp/session/ClassName
--cct FILE            Annotated CCT pickle; inferred from --csc-result-dir when possible
--class NAME          Class name used to infer <ClassName>_cct.pkl
--output FILE         Report JSON path; defaults to cct_failure_localization.json
--dot-output FILE     DOT risk-view path; defaults to cct_failure_localization_<strategy>.dot
--edge-strategy NAME  Edge strategy for compatibility ranking and DOT view
--list-strategies     Print available edge strategies and exit
--no-statement-aware  Do not read testcases.json traces; only use cct_only
--render-cct          Render the DOT risk view during this run
--cct-formats LIST    Comma-separated render formats for --render-cct
--top-k N             Keep only top N records in each ranking

The report contains node rankings and one or more interval edge strategy rankings:

  • condition_node_ranking: risk that a condition expression itself is faulty.
  • interval_rankings.cct_only: CCT subtree failure evidence only.
  • interval_rankings.statement_presence: keeps edge risk only when the concrete edge segment contains ASSIGN or RETURN trace events.
  • interval_rankings.edge_divergence_gated: Edge-level Divergence and Gated Evaluation; empty statement regions receive score 0, while non-empty regions use base_risk * (1 + log10(1 + LC)), where LC is the number of distinct ASSIGN/RETURN source lines.
  • condition_interval_ranking: compatibility alias for the default interval strategy.

Infeasible, out-of-range, empty, skipped, and unsupported paths are excluded from the executable denominator. A node or interval is ranked only when its subtree contains at least one Refined TBFV failure. With --csc-result-dir, the default interval strategy is statement_presence because testcases.json provides trace paths. With --cct only, the default interval strategy falls back to cct_only.

The DOT risk view highlights suspicious condition nodes and condition-interval edges. It is generated as text by default and named by strategy, such as cct_failure_localization_statement_presence.dot. Render it explicitly with --render-cct or with the session-level render_cct.sh script.

Aggregating Failure-Localization Rankings

The raw localization report ranks dynamic CCT nodes and edges. For automated fault-localization evaluation, aggregate those dynamic rankings into source-level locations before computing top-k metrics: 

python3 failure_localization_aggregate_tool.py \
  --report csc_tmp/my_session/MyClass/cct_failure_localization.json \
  --source-file dataset/CSC_V2_dataset/MySubject/MyClass.java \
  --output csc_tmp/my_session/MyClass/cct_failure_localization_aggregated.json

The aggregated report contains aggregated_condition_node_ranking and aggregated_interval_rankings. Condition nodes are grouped by source line and condition text. Statement-aware edge rankings are grouped by the condition transition: from_line, from_condition, outcome, to_line, and to_condition.

For statement_presence and edge_divergence_gated, fault-localization evaluation should use statement_lines collected from the trace segment as the predicted source locations. The line_interval and condition_anchor_span fields are condition anchors, not necessarily the executed statement region, especially in loops.

15. FAQ

Q: What's the difference between branch coverage and CSC?

Branch coverage checks each static branch (the True and False sides of every if/while). CSC checks each dynamic occurrence of a condition along execution paths, distinguished by loop count. This means i > 1 in iteration 1 and i > 1 in iteration 2 are treated as separate conditions, providing deeper coverage than branch coverage but avoiding full path explosion.

Q: When should I use Expanded CSC over Original CSC?

Use Expanded CSC when your program has loops where the number of iterations depends on integer inputs within a known range. The bounded-range check prevents infinite loop unrolling while still exploring meaningful variations. Use Original CSC for programs where loop iterations are fixed or don't depend on input values in interesting ways.

Q: Why does path truncation happen?

When a condition's loop_count exceeds MAX_LOOP_BOUND (2), the path is cut and stored as a truncated leaf. This prevents the CCT from growing unboundedly for programs with many loop iterations. In Expanded CSC mode, truncated leaves can be "promoted" to internal nodes if in-range Z3 solutions exist for continuing exploration.

Q: How do I interpret the DOT visualization?

  • Internal nodes show (loop_count) condition_string plus the WP constraint
  • Solid green edges = True branch; dashed red edges = False branch
  • Truncated leaves (orange) mean the path was cut at MAX_LOOP_BOUND
  • Expanded nodes/leaves (blue) were created during bounded-range expansion
  • Infeasible leaves (red) were proven UNSAT by Z3

Q: Can I use the CSC engine without Java?

Yes. The core CCT data structure and CSC algorithms are language-agnostic. You can build CCTs directly from CSC path strings (see Section 11.1). The Java bridge is optional and only needed when you want to automatically instrument and execute Java programs.

Q: What happens when Z3 returns "OK"?

"OK" means UNSAT — the path constraint has no satisfying assignment. The branch is then marked as infeasible ("X") in the CCT, which prevents the algorithm from trying to explore it again.

Q: Why do some conditions appear with "!(...)" wrapping in the trace but not in the CCT?

The Java auto-instrumenter synthesizes negated conditions like !(x < 0) for the else- branch of an if-statement. The CSC engine normalizes these back to their original form (x < 0) with the result flipped, so that both the if-true and if-false paths use the same condition string in the CCT. This ensures ancestor-loop matching works correctly.

Q: How are multi-method programs handled?

The PARAM_MAP mechanism in execution traces records how caller arguments map to callee parameters. The WP substitution engine resolves these bindings to build correct path constraints that span method call boundaries.
