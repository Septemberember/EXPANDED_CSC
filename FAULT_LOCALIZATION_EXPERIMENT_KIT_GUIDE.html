Fault Localization Experiment Kit Guide

This guide explains how to run the portable dataset-level wrapper for CSC, Refined TBFV, CCT-based failure localization, aggregation, and evaluation. The wrapper preserves the existing input and output formats.

The package is portable as part of the CSC_EXPANDED project. Copying only the runner script is not enough because it reuses the current CSC, TBFV, localization, Java bridge, and bundled SFL baseline implementation.

Environment

Install or verify these dependencies on the target machine:

  • Python 3.10 or newer
  • JDK with java and javac
  • Maven, if the Java bridge must be rebuilt
  • z3-solver Python package
  • numpy Python package, required by the optional SFL baseline
  • Graphviz dot, optional for rendering CCT images
pip install -r fault_localization_experiment_kit/requirements.txt
python run_fault_localization_experiment.py --check-env
python run_fault_localization_experiment.py --build-java-bridge

The manifest schema is the same one used by validate_fault_localization_dataset.py and evaluate_fault_localization.py.

Full Experiment

python run_fault_localization_experiment.py \
  --dataset-root dataset/EX_CSC_dataset \
  --experiment-root experiments/FL-EX_CSC_dataset \
  --session-prefix fl_061002 \
  --mode expanded \
  --range-bound 200 \
  --strategy batch \
  --workers 4 \
  --max-iter 100

The default behavior runs mutant programs only. Add --include-original if original programs should also be executed.

Optional SFL Baseline

Add --run-sfl-baseline to compute the line-level SFL baseline from the same archived traces and Refined TBFV pass/fail labels. This stage runs after CCT localization and artifact archiving, and writes its own row-level files under baseline-SFL/. 

python run_fault_localization_experiment.py \
  --dataset-root dataset/EX_CSC_dataset \
  --experiment-root experiments/FL-EX_CSC_dataset-sfl \
  --session-prefix fl_061003 \
  --mode expanded \
  --range-bound 200 \
  --strategy batch \
  --workers 4 \
  --max-iter 100 \
  --run-sfl-baseline

SFL rankings are recomputed by default to avoid stale or top-k-truncated reports. Use --reuse-existing-sfl only when intentionally reusing existing per-mutant sfl_localization.json files. The runner first looks for the bundled baseline at CSC_EXPANDED/baseline/run_sfl_baseline_experiment.py, then falls back to the older source-checkout layout at CSC_EXT/baseline/.

Selective Runs

python run_fault_localization_experiment.py \
  --dataset-root dataset/EX_CSC_dataset \
  --experiment-root experiments/FL-debug \
  --session-prefix fl_debug \
  --subjects GradePolicy,InventoryReorder \
  --mutants GradePolicy_M1,InventoryReorder_M2 \
  --resume

Important Options

OptionMeaning
--dataset-rootDataset root containing subject directories and the mutant manifest.
--experiment-rootDirectory where experiment reports and archived artifacts are written.
--session-prefixPrefix for generated csc_tmp sessions and summary discovery.
--mode expandedBoundary-enabled CSC mode used in current experiments.
--range-boundBoundary exploration range bound.
--strategy batchBatch test-case generation strategy.
--workersNumber of worker processes used by batch generation.
--max-iterMaximum CSC expansion iterations per program.
--run-sfl-baselineRun the line-level SFL baseline after CCT localization and write baseline-SFL outputs.
--reuse-existing-sflReuse existing per-mutant SFL reports instead of recomputing rankings.
--resumeSkip mutants that already have matching evaluation reports under csc_tmp.

Output Compatibility

The runner writes the same key files as the manual workflow: fault_localization_rows.jsonl, fault_localization_rows.csv, and fault_localization_summary.md. The archived artifacts/csc_tmp directory preserves per-mutant intermediate files for later inspection.

When --run-sfl-baseline is used, the runner also writes baseline-SFL/sfl_fault_localization_rows.jsonl, baseline-SFL/sfl_fault_localization_rows.csv, and baseline-SFL/sfl_fault_localization_summary.md. These SFL rows use a compatible row-level schema for later cross-dataset aggregation.

Post-Experiment Reporting Check

A complete fault-localization report should include an overall category-agnostic table and two fault-category tables: condition/control-flow mutants and statement/data-flow mutants. The overall table reflects the practical unknown-fault-type workflow, while the split tables explain which CCT view is responsible for each fault mechanism.

  • Condition/control-flow mutants should show the CCT composite and condition-node view.
  • Statement/data-flow mutants should show the CCT composite and edge-divergence-gated interval view.
  • If fault_localization_summary.md only has the overall table, regenerate the summary from fault_localization_rows.jsonl or fault_localization_rows.csv; the CSC/TBFV runs do not need to be repeated.
